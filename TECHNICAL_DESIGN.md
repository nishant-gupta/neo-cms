# Lightweight, Cost-Effective CMS on Google Cloud

Diagrammatic views are available in [ARCHITECTURE_DIAGRAMS.md](./ARCHITECTURE_DIAGRAMS.md).

## 1) Goals and Constraints

### Primary goals
- Provide a lightweight CMS that is cheap to run and easy to maintain.
- Store all media assets (images, videos, docs, PDFs) in Google Cloud Storage (GCS).
- Allow content authoring in Google Docs (simple editor for non-technical users).
- Support preview and publish workflows for website pages.
- Deliver very high performance and SEO with a target of Lighthouse 95-100.

### Non-goals (v1)
- Full WYSIWYG page builder with arbitrary drag-and-drop layouts.
- Complex multi-language workflow automation.
- Real-time collaborative editing beyond Google Docs capabilities.

---

## 2) Recommended Architecture (v1)

Use a **headless CMS pipeline** with static page generation:

- **Authoring**: Google Docs (one doc per page/article).
- **Asset storage**: Google Cloud Storage bucket(s).
- **Metadata + workflow state**: Firestore (or a single JSON config in GCS for micro scale).
- **Ingestion and transform**: Cloud Run service + Cloud Tasks (for async jobs).
- **Site generation**: Eleventy (11ty) static generator using Nunjucks + Markdown/JSON data.
- **Hosting**: Cloud Storage static hosting + Cloud CDN **or** Firebase Hosting (on GCP).
- **Preview**: preview subdomain deployed from draft artifacts.
- **Publish**: promote validated build to production bucket/hosting target.

This architecture keeps runtime complexity low: mostly static files, event-driven jobs, and managed services.

---

## 3) Why this stack is lightweight and low-maintenance

1. **Google Docs as CMS editor**
   - No custom editor UI needed.
   - Authors already know it.
   - Minimal training and support.

2. **Static site output**
   - Very low hosting cost.
   - No server-rendering runtime by default.
   - Easier Lighthouse optimization.

3. **Managed GCP services**
   - Cloud Run (serverless, scales to zero).
   - Firestore (no ops database).
   - Cloud Storage + CDN for inexpensive global delivery.

4. **Simple frontend**
   - Vanilla HTML/CSS + tiny JS (Alpine.js optional).
   - Avoid heavy frameworks and hydration costs.

## 4) High-Level Component Design

## A. Authoring and Content Model

### Authoring in Google Docs
- Authors write content in predefined Google Docs templates:
  - `Title`
  - `Summary`
  - `Body` (headings, paragraphs, tables, lists)
  - Optional callouts (convention via heading markers/tags)
- Each doc is mapped to a page via `docId`.

### Metadata model
- Keep metadata outside doc body for clean publishing:
  - `slug`
  - `status` (draft, review, approved, published)
  - `publishAt`
  - `author`
  - `tags`
  - `seoTitle`, `seoDescription`
  - `canonicalUrl`
  - `socialImage`

Store metadata in Firestore collection:
- `pages/{pageId}`
- `assets/{assetId}`
- `revisions/{revisionId}`

## B. Assets on Google Cloud Storage

### Bucket strategy
- `cms-assets-prod`: original uploaded assets.
- `cms-assets-derivatives`: optimized images/web formats.
- `cms-site-preview`: preview static build.
- `cms-site-prod`: published static build.

### Asset processing
- Upload assets through signed URLs generated by Cloud Run API.
- Trigger Cloud Run job on object finalize event:
  - Extract metadata (size, mime, dimensions if image/video).
  - Generate optimized versions:
    - images: AVIF/WebP + responsive widths
    - video: poster thumbnail
  - Write asset metadata to Firestore.

## C. Content Ingestion Pipeline

### Flow
1. User marks document as ready for preview in CMS admin.
2. Cloud Run calls Google Docs API for doc structure.
3. Parser transforms doc to sanitized HTML/Markdown AST.
4. Internal normalizer maps doc sections to template data.
5. Build job generates static page(s) into preview bucket.
6. Preview URL returned to editor.

### Parser constraints
- Support only a controlled set of doc features in v1:
  - headings, paragraphs, lists, links, images, tables, quotes.
- Strip unsupported styles to avoid invalid HTML and performance issues.

## D. Preview and Publish

### Preview
- Every preview gets immutable revision path:
  - `https://preview.example.com/rev/{revisionId}/slug/`
- Reviewers can verify content and SEO tags.

### Publish
- Publish action promotes selected revision from preview build to prod.
- Two options:
  1. Rebuild with pinned revision snapshot and deploy to prod bucket.
  2. Copy verified artifact bundle from preview to prod.

Recommended: **option 1** for deterministic reproducible release with audit log.

## 5) Frontend and Lighthouse Strategy

## Framework choice
- Use **Eleventy (11ty)** with Nunjucks + Markdown/JSON data.
- CSS: hand-written, utility-lite approach, no heavy runtime CSS framework.
- JS: zero JS by default; only progressive enhancement where needed.

## Performance requirements
- LCP <= 2.5s on mobile, CLS < 0.1, INP < 200ms.
- Page weight target:
  - HTML: < 40 KB gzipped
  - CSS: < 20 KB gzipped critical path
  - JS: < 30 KB gzipped total

## Implementation tactics for high Lighthouse
- Inline critical CSS, defer non-critical CSS.
- Preload primary hero image and fonts (if custom fonts are used).
- Use system fonts where possible for zero font payload.
- Responsive images (`srcset`, `sizes`, AVIF/WebP fallback).
- Lazy-load below-the-fold media.
- Use strict cache-control headers with immutable fingerprints.
- Generate static Open Graph/Twitter/JSON-LD tags at build time.
- Eliminate render-blocking JS; use `defer` everywhere.
- Accessibility checks in CI (aria labels, color contrast, semantics).

---

## 6) Security and Access Control

- Google OAuth for admin UI (Cloud Identity / Google Workspace users).
- IAM roles:
  - Editors: create/update drafts and assets.
  - Publishers: approve and publish.
  - Admins: manage templates and settings.
- Signed URLs for direct upload to GCS; no public write access.
- GCS buckets private by default; only site output bucket is publicly readable.
- Sanitize all content transformed from Google Docs before render.
- Enable Cloud Audit Logs for publish operations.

## 7) Suggested API Surface (Cloud Run)

Base path: `/api/v1`

- `POST /pages` create page metadata
- `PUT /pages/{id}` update metadata
- `POST /pages/{id}/sync-doc` import latest Google Doc
- `POST /pages/{id}/preview` create preview revision/build
- `POST /pages/{id}/publish` publish selected revision
- `POST /assets/signed-upload-url` request signed upload URL
- `GET /assets/{id}` read asset metadata
- `GET /revisions/{id}` revision details and status

Asynchronous tasks:
- `buildPreview(revisionId)`
- `buildProd(revisionId)`
- `processAsset(objectPath)`

## 9) Deployment Topology

## Environments
- `dev`: feature testing, internal users.
- `staging`: pre-production validation.
- `prod`: public site.

## CI/CD
- GitHub Actions (or Cloud Build):
  1. Lint + unit tests.
  2. Build static site.
  3. Run Lighthouse CI on representative pages.
  4. Deploy preview/prod artifacts to corresponding bucket/hosting target.

## Rollback
- Keep previous production artifact snapshots.
- Rollback by switching CDN origin path/version pointer.

## 10) Cost Optimization Guidelines

- Prefer static serving over dynamic SSR for most pages.
- Use Cloud Run min instances = 0 for ingestion/admin APIs.
- Process media asynchronously and only generate needed derivatives.
- Set lifecycle policies on old previews and stale revisions.
- Use CDN caching aggressively for static content.
- Monitor top cost drivers:
  - GCS storage + egress
  - CDN egress
  - Cloud Run CPU/memory seconds

Expected small-to-medium setup can remain low cost due to static hosting and scale-to-zero backend.

## 11) Implementation Plan (Phased)

## Phase 0 - Discovery (1 week)
- Finalize content types and template set.
- Confirm Google Workspace permissions and OAuth setup.
- Define Lighthouse SLOs and target pages.

## Phase 1 - Core platform (2 weeks)
- Provision GCS buckets, Firestore, Cloud Run services.
- Implement auth, page metadata APIs, signed asset upload flow.
- Build initial Google Docs parser and sanitization.

## Phase 2 - Static site + preview (2 weeks)
- Build Eleventy templates and asset helpers.
- Implement revisioned preview builds and preview URLs.
- Add editorial preview workflow in admin UI.

## Phase 3 - Publish + governance (1-2 weeks)
- Implement publish approval flow and audit logs.
- Add rollback controls and revision history.
- Enforce role-based access controls.

## Phase 4 - Performance and hardening (1 week)
- Lighthouse CI gates in pipeline.
- Security review (content sanitization, IAM, bucket policies).
- Observability dashboards and alerts.

## 12) MVP Scope (Recommended)

Ship the smallest useful v1:
- 3 page templates (Landing, Article, Basic Content).
- Google Docs import for headings/paragraphs/lists/images/tables.
- Asset upload + automatic image optimization.
- Preview and publish workflow.
- Lighthouse CI checks for 3 key routes.

Defer advanced features (multi-language, personalization, workflow automation) to v2.

---

## 13) Risks and Mitigations

1. **Google Docs formatting inconsistency**
   - Mitigation: strict template guide + parser whitelist + validation warnings.

2. **Media bloat hurting performance**
   - Mitigation: automatic compression, responsive variants, upload limits.

3. **Uncontrolled publish process**
   - Mitigation: approval roles, immutable revisions, audit trails.

4. **SEO regressions**
   - Mitigation: template-level required SEO fields + CI Lighthouse checks.

---

## 14) Success Metrics

- Lighthouse (mobile) >= 95 across key pages; target 100 for static content pages.
- Median publish time < 2 minutes from approval.
- Zero critical security findings in quarterly review.
- Monthly infrastructure cost within predefined budget threshold.
- <1% failed preview/publish jobs over 30 days.
